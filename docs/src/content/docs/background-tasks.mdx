Background tasks let you dispatch long-running coding work to an AI agent runner — either an external CLI tool (Claude, Codex, Kimi, Gemini, Opencode, or Cursor) or clawq's own in-process `local` runner. Tasks can run in a git worktree isolated from your main branch. The task runs asynchronously while clawq tracks its state, captures logs, and optionally rebases and fast-forward merges the result back when complete.

## delegate vs background

`clawq delegate` is the high-level shortcut. It picks a runner automatically, resolves the repo, and queues a single task:

```bash
clawq delegate "Refactor the scheduler module to use a priority queue"
```

`clawq background add` gives you full control — explicit runner, model, repo, and branch:

```bash
clawq background add claude --model anthropic:claude-sonnet-4-6 \
  ~/src/myproject --branch feature/scheduler-refactor \
  "Refactor the scheduler module to use a priority queue"
```

Use `delegate` for quick ad-hoc tasks and `background add` when you need to specify the runner or branch explicitly.

## Runners

Seven runners are supported. Six are external CLI tools; one (`local`) runs in-process. The daemon selects from available external runners when none is specified:

| Runner | Binary | Session ID Strategy | Auto Order |
|--------|--------|---------------------|------------|
| `claude` | `claude` | Pre-generated UUID | 1st |
| `codex` | `codex` | Parse JSONL field `thread_id` | 2nd |
| `kimi` | `kimi` | Parse log regex `Session=...` | 3rd |
| `gemini` | `gemini` | Index-based | 4th |
| `opencode` | `opencode` | Parse log regex `session=ses_...` | 5th |
| `cursor` | `cursor-agent` | Parse log regex `chat...=...` | 6th |
| `local` | (in-process) | Stable `__bg_task:<id>` session | N/A |

External runners are invoked with their respective `--yolo`/`--dangerously-*` flags to enable non-interactive operation. Runner definitions live in `src/runner_framework.ml`.

### Local Runner

The `local` runner executes agent turns in-process using clawq's own LLM provider and tool system. No external binary is needed. Local tasks:

- Write logs to `~/.clawq/background-logs/task-<id>.log` (preamble, agent history updates, and final result)
- Persist real session history under the stable session key `__bg_task:<id>`, so follow-up messages resume the same conversation
- Support **cooperative cancellation** — `background cancel` sets an interrupt flag checked between agent turns; the task stops cleanly and is marked Cancelled
- Use **log-file freshness** for health diagnosis — stall detection checks the log modification time, not just elapsed wall-clock time
- Can be bound to an agent template via `--agent <name>`, or use the default session turn

### Hosted Runner Isolation

External runners can be isolated from the host when
`security.hosted_runner_isolation` is `"prefer"` or `"require"`: a minimal
allowlisted environment (credentials stripped by construction) plus a
bubblewrap/firejail sandbox restricted to the task worktree, log directory,
and provider auth directories. `"require"` fails task startup closed when no
sandbox backend is available. See `docs/hosted-runner-isolation.md`.

### Native Subagents

`clawq subagents` is the human-friendly wrapper for native/local background tasks:

```bash
clawq subagents start --agent coder --model xiaomi-token-plan-sgp:mimo-v2.5-pro \
  ~/src/myproject "Investigate and fix the failing scheduler test"
clawq subagents list
clawq subagents send 42 "Please include the exact failing command"
clawq subagents transcript 42 --regex scheduler
clawq subagents stop 42
```

These commands use the same background task table as `clawq background`. The compatibility aliases `background start`, `background stop`, `background send`, and `background transcript` route to the same behavior.

Task-tree entries can also carry native-agent execution metadata. The `task_tree` tool accepts `depends_on`, `agent_model`, `agent_type`, `agent_prompt`, `agent_details`, and `autostart` fields on task add/update operations. `task_start_agent` starts a task as a native/local subagent using that metadata. When `autostart` is true, clawq queues the task automatically after every listed dependency is done or cancelled.

## Worktree Lifecycle

When a task is created with a branch name (or when `delegate` creates one automatically), clawq provisions a git worktree:

1. A new branch is created off the current HEAD of the repo
2. A worktree is checked out at `~/.clawq/background-worktrees/task-<id>/`
3. The runner is launched with the worktree as its working directory
4. Logs are written to `~/.clawq/background-logs/task-<id>.log`
5. On completion, the worktree can be rebased, fast-forward merged, and removed via `background finalize`

Tasks without a branch name run in the repo root directly and have no worktree.

```bash
# Show worktree path for a task
clawq background show 42
```

## Session ID Tracking

Each runner uses a different mechanism to capture its session ID for later resume:

- **Claude**: The session ID is pre-generated as a UUID by clawq before launching the process and passed via `--session-id`. This ID is known before the runner starts.
- **Codex**: The JSONL output is scanned for the first JSON object containing a `thread_id` field.
- **Kimi**: The log output is scanned with the regex `Session[=: ]+([^ \t\n\r]+)`.
- **Opencode**: The log output is scanned with the regex `session[=: ]+(ses_[^ \t\n\r]+)`.
- **Cursor**: The log output is scanned with the regex `chat[A-Za-z]*[=: ]+([^ \t\n\r]+)`.
- **Gemini**: Uses index-based tracking (no session ID extracted from output).

The captured session ID is stored in the `background_tasks` table and used for resume operations.

## Runner Question Relay

Remote runners can ask the user clarifying questions during execution. When a background task is spawned, the daemon automatically sets three environment variables in the runner process:

| Variable | Purpose |
|----------|---------|
| `CLAWQ_RUNNER_TOKEN` | Bearer token for authenticating to the gateway |
| `CLAWQ_MCP_URL` | MCP-over-HTTP endpoint (e.g. `http://127.0.0.1:8080/mcp`) |
| `CLAWQ_RUNNER_ASK_URL` | Simple REST endpoint (e.g. `http://127.0.0.1:8080/runner/ask`) |

Runners that support MCP can discover and call `ask_user_question` via standard `tools/list` + `tools/call` on the MCP URL. Non-MCP runners can POST directly to the REST URL:

```bash
curl -X POST http://127.0.0.1:8080/runner/ask \
  -H "Authorization: Bearer $CLAWQ_RUNNER_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"questions": [{"type": "text", "question": "Which database should I use?"}]}'
```

The question is relayed to the user's active channel (Telegram, Discord, Slack, web) and the endpoint blocks until the user answers or the timeout expires (default 300s). Tokens are localhost-only and expire after 24 hours by default.

See [Tools > MCP-over-HTTP for Remote Runners](/docs/tools/#mcp-over-http-for-remote-runners) for endpoint details and configuration.

## Resume and Message

Once a task has started and a session ID is captured, you can send follow-up messages to the running agent:

```bash
# Resume a paused/stopped task using the runner's native session support
clawq background resume 42

# Send a message and resume (injects the message into the agent's conversation)
clawq background message 42 "Also update the test suite"
```

For external runners, `background resume` uses `By_session_id` mode when a session ID was captured, or `By_last` (resume most recent session) as fallback. For `local` tasks, messages resume the stable `__bg_task:<id>` session directly.

`background message` appends the message to the runner's prompt and resumes, letting you steer an in-progress task without losing context.

For agents running in unrelated live sessions, use `clawq session send <session_key> ...` or the `send_to_session` tool. Background stop/cancel semantics apply to background task IDs only.

## Transcripts

Use `subagents transcript <id>` or `background transcript <id>` to inspect bounded task conversation history:

```bash
clawq subagents transcript 42 --regex "failing test" --max-lines 100
clawq background transcript 42 --export
```

Transcript lookup prefers the stable local session history for `__bg_task:<id>`, then ACP history, then the captured log file. Regex filtering is applied before size checks. Inline output defaults to 200 lines and hard-caps at 300 lines; oversized results are refused inline and exported as searchable JSONL under the clawq state directory.

## Completion Pass

When any worktree-backed task finishes (Succeeded or DirtyWorktree), the system sends a **completion pass** message to the agent before finalizing:

1. The agent is resumed with its existing session context and a verification message
2. The agent commits any remaining changes, rebases against `master`, runs checks, and outputs the sentinel `OK_TASK_DONE_CHECKED_REBASED_COMMITED`
3. The system detects the sentinel and proceeds to finalization

The completion pass also includes a **review-and-fix guard**: the agent must review all changes (`git diff master..HEAD`) for correctness, quality, completeness, and safety before signaling done. Issues found must be fixed and re-reviewed until clean.

This ensures every worktree task goes through both verification and review, regardless of whether automerge is enabled.

Parent sessions receive a concise completion wake message with a bounded result preview and a pointer such as `subagents transcript 42`; full logs and transcripts are not injected into the parent context.

## Automerge

Automerge is **enabled by default**. When a task is queued with automerge (the default for both `delegate` and `background_task_enqueue`), clawq automatically attempts to rebase and fast-forward merge the worktree branch into the repo's main branch after the completion pass succeeds:

1. The branch is rebased onto the current HEAD of the target branch
2. If rebase succeeds, the target branch is fast-forward merged and the worktree is removed
3. If conflicts are detected, the task is marked with a merge conflict status

To manually finalize after automerge failure or when automerge was disabled (`automerge: false`):

```bash
clawq background finalize 42
```

This performs the same rebase + fast-forward merge + cleanup sequence interactively.

## Recovery

If a task fails, gets stuck, or a runner crashes mid-task, use `background recover` to create a new task with the same context but a fresh invocation:

```bash
# Recover with the same runner
clawq background recover 42

# Recover with a different runner
clawq background recover 42 --runner claude

# Recover with a different model
clawq background recover 42 --model anthropic:claude-opus-4-6
```

Recovery loads the original task's prompt, repo path, and branch, then queues a new task. The original task is marked as superseded. Tasks support up to 3 automatic retries before requiring manual intervention.

<Callout type="info">
`background retry` re-queues a failed task unchanged (max 3 retries). `background recover` lets you override the runner or model when retrying — useful when a specific runner is consistently failing on a task.
</Callout>

## Full Command Reference

```bash
clawq background list                              # List all tasks
clawq background show <id>                         # Show task details
clawq background add <runner> [--model M] <repo>   # Queue a task
              [--host <direct|herdr>] [--branch <name>] <prompt>
clawq background wait <id> [--timeout <secs>]      # Wait for completion
clawq background logs <id> [--lines N]             # Show log lines
              [--offset N] [--follow]
clawq background transcript <id> [--regex R]       # Show bounded transcript
              [--max-lines N] [--export]
clawq background start <runner> [--model M] <repo> # Alias for add
              [--branch <name>] <prompt>
clawq background send <id> <message...>            # Alias for message
clawq background stop <id>                         # Alias for cancel
clawq background resume <id>                       # Resume a task
clawq background message <id> <message...>         # Send message and resume
clawq background cancel <id>                       # Cancel a task
clawq background retry <id>                        # Re-queue failed task
clawq background recover <id> [--runner R]         # Recover with new runner
              [--model M]
clawq background finalize <id>                     # Rebase, fast-forward merge, clean up
clawq subagents list                               # List native/local tasks
clawq subagents start [--model M] [--agent NAME]   # Queue a local task
              <repo> <prompt>
clawq subagents send <id> <message...>             # Message a local task
clawq subagents transcript <id> [--regex R]        # Read bounded transcript
clawq subagents stop <id>                          # Cancel a local task
```

## Plan Pipeline

`clawq plan` is a higher-level abstraction that chains multiple background agent invocations into a structured pipeline:

1. **Planner** — an agent turn that produces a written plan for the goal
2. **Plan review** — a review agent that critiques and refines the plan (up to `max_plan_review_iters` iterations)
3. **Coder** — a background runner that implements the plan in a worktree
4. **Code review** — a review agent that checks the implementation (up to `max_code_review_iters` iterations)

```bash
# Start a plan pipeline
clawq plan start "Add pagination to the sessions list endpoint"

# With options
clawq plan start "Add pagination" --repo ~/src/myproject --runner claude

# Check progress
clawq plan show <id>

# View stage logs
clawq plan logs <id>

# Cancel
clawq plan cancel <id>
```

Each pipeline stage runs sequentially. The plan is written to a file (shown in `plan start` output) before the coder stage begins, so you can inspect or edit it before coding starts.

<Callout type="info">
Use `delegate` for single-shot tasks and `plan` when the goal is complex enough to benefit from an explicit planning and review loop.
</Callout>

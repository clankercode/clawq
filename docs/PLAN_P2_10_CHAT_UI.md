# P2-10: Chat UI

## Context

nullclaw has a companion Chat UI (Svelte-based) with streaming responses, a tool execution timeline, and approval gates for high-risk actions. clawq has no frontend. This plan adds a minimal but production-quality web chat UI served directly from the clawq HTTP gateway.

## Design Philosophy

**Served from the gateway**: the UI is static HTML/CSS/JS embedded in the binary (as OCaml string literals from build-time files), served at `GET /` from `http_server.ml`. No separate server process.

**Framework choice**: Vanilla JS or minimal framework (no build step required). This avoids a Node.js/npm dependency in the build pipeline. The UI is a single `index.html` + `chat.js` + `chat.css`.

**Features** (prioritized):
1. Streaming chat (SSE from `/chat/stream`)
2. Message history display
3. Tool execution timeline (events from `/events` or inline in SSE stream)
4. Markdown rendering (marked.js CDN or embedded)
5. Code syntax highlighting (highlight.js CDN or embedded)
6. Approval gates for high-risk tool invocations (requires P2-08 policy work first)
7. Pairing flow (enter 6-digit code, store token in localStorage)

## Architecture

### Backend Changes

**New SSE stream endpoint** (already partially exists as `/chat/stream`):

Extend to include tool events inline in the SSE stream:
```
data: {"type":"text","content":"Hello "}
data: {"type":"text","content":"world"}
data: {"type":"tool_start","tool":"execute_shell","args":{"command":"ls"}}
data: {"type":"tool_result","tool":"execute_shell","success":true,"preview":"file1.txt\nfile2.txt"}
data: {"type":"done"}
```

This embeds tool timeline data directly in the chat stream rather than requiring a separate `/events` endpoint.

**New endpoint: `GET /`** — serve `index.html` (embedded as OCaml string constant).

**New endpoint: `GET /chat.js`** — serve JavaScript.

**New endpoint: `GET /chat.css`** — serve CSS.

### Frontend: `static/index.html`, `static/chat.js`, `static/chat.css`

**UI components:**
- **Chat thread**: scrollable message list with user/assistant/tool-result distinction
- **Input box**: textarea with Enter-to-send (Shift+Enter for newline)
- **Tool timeline**: collapsible per-turn panel showing tool calls and results
- **Status bar**: connection state, model name, token count (if available)
- **Pairing modal**: shown if initial request returns 401/403, prompts for 6-digit code

**Streaming:**
```javascript
const es = new EventSource('/chat/stream?message=' + encodeURIComponent(input));
es.onmessage = (e) => {
  const event = JSON.parse(e.data);
  if (event.type === 'text') appendText(event.content);
  else if (event.type === 'tool_start') addToolStart(event);
  else if (event.type === 'tool_result') addToolResult(event);
  else if (event.type === 'done') es.close();
};
```

**Session:** The browser uses `sessionStorage` for the session key (cleared on tab close). A "New conversation" button clears it.

**Auth:** Bearer token stored in `localStorage['clawq_token']`. Included in all API requests as `Authorization: Bearer {token}`. If missing or rejected, show pairing modal.

**Markdown:** Use `marked.js` loaded from CDN (with `integrity` hash). Fallback: display as preformatted text.

### Embedding Strategy

Build-time step: read `static/index.html`, `static/chat.js`, `static/chat.css` → generate `src/chat_ui_assets.ml` with string constants:

```ocaml
(* generated by scripts/embed_ui.sh *)
let index_html = {|<!DOCTYPE html>...|}
let chat_js = {|const es = ...|}
let chat_css = {|body { ... }|}
```

Makefile target `make embed-ui` runs the generation script. Generated file is checked into git (same pattern as `src/extracted/clawq_core.ml`).

Alternative: use dune's `(embed_in_module ...)` stanza (dune 3.0+) to embed files directly without a script.

### Tool Timeline Format

Each agent turn collects tool events. The existing `audit.ml` already logs `ToolInvocation` and `ToolResult`. For the UI, emit them inline in the SSE stream from `agent.ml`'s `turn_stream` callback:

```ocaml
(* In agent.ml turn_stream *)
on_chunk (Printf.sprintf {|{"type":"tool_start","tool":"%s","args":%s}|} name args_json);
(* ... invoke tool ... *)
on_chunk (Printf.sprintf {|{"type":"tool_result","tool":"%s","success":%b,"preview":"%s"}|}
           name success (String.escaped preview));
```

### Approval Gates (Deferred)

Tool approval requires the policy system from P2-08. For now, the UI can display tool invocations but does not block them for approval. Add a `requires_approval` field to the tool event for future use.

## Test Strategy

1. **Asset serving**: `GET /` returns 200 with `Content-Type: text/html`
2. **JS/CSS serving**: correct content types
3. **SSE tool events**: mock agent turn with tool call → stream includes `tool_start` and `tool_result` events
4. **Auth**: request without token to protected endpoint → 401
5. **Pairing modal trigger**: frontend behavior (manual/browser test)
6. **Streaming parse**: unit test SSE line parser in `http_server.ml`

## Dependencies

- No new opam packages
- CDN resources: `marked.js`, optionally `highlight.js` (loaded in browser, not build dep)
- Build script: `scripts/embed_ui.sh` (bash)
